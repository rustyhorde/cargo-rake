# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A configuration-driven build tool that runs named targets declared in a `Rakefile.toml`, in dependency order. Each target owns an ordered array of named commands (`[[target.<t>.command]]`, each with a `name`, one *kind* of body, and an optional `skip_on_error` flag), an optional `depends_on` list, and an optional `tools` list. A command body is either a `cmd` of program + args spawned directly (no shell) **or** one or more shell variants — `sh`/`fish`/`ps` — each a command line run through that shell (`sh -c`, `fish -c`, PowerShell `-Command`) so it expands `$(...)`/`~`/`$VAR`/globs. At run time rake **auto-detects the current shell** (see `detect_shell_family`) and runs the matching variant; selection is **strict** — if the detected shell has no matching variant the run aborts (`MissingShellVariant`). `cmd` is mutually exclusive with the shell variants, the variants may coexist, and a command with no body at all is a validation error. There is no top-level shell key. A target's commands run in array order; `skip_on_error` (default `false`) lets a command's non-zero exit be tolerated so the target's remaining commands and the dependency chain continue. The dependency graph is at the target level.

An **optional** top-level `toolchain` key (a single token, e.g. `stable`/`beta`/`nightly`, or any rustup toolchain like `1.95.0` / `nightly-2024-01-01`) names the Rust toolchain channel the project targets. **Both** binaries act on it, but **only when it is explicitly present** — a Rakefile without the key is a no-op (commands use whatever toolchain is active). When the key is set, before running targets the binary ensures the channel is available: if `cargo` is absent it offers (interactively) to install Rust via the official rustup installer (`curl … sh.rustup.rs | sh -s -- -y --default-toolchain <ch>`), prepends `~/.cargo/bin` to `PATH`, and continues; if `cargo` is present but the channel is not installed it offers to `rustup toolchain install <ch>`; once available it pins the run via `RUSTUP_TOOLCHAIN=<ch>` so spawned commands use that channel. With no interactive terminal it prints the requirement and exits non-zero rather than hanging; without rustup (e.g. a distro-packaged cargo) it accepts the existing `cargo` as-is. The bootstrap-from-nothing branch (no `cargo` at all) only realistically matters for the standalone `rake` binary, since `cargo rake` is always invoked through an existing cargo; when the key is omitted, `rake` still bootstraps `stable` if no `cargo` exists.

A target's `tools` names entries in a top-level `[tool]` table split into two categories: `[tool.cargo.<name>]` (cargo-installable tools — cargo subcommands, etc.) and `[tool.os.<name>]` (OS-level dependencies like `docker`/`pkg-config` that rake cannot `cargo install`). The two categories share one flat reference namespace (a target's `tools` names resolve against both; a name must be unique across them, else `DuplicateTool`). Tools are ensured lazily — only when a target that references the tool is actually run, and at most once per run; parse-time validation never executes a `check`. For a **cargo** tool: its `check` command probes for presence (zero exit = installed; for a cargo subcommand use `["cargo", "<sub>", "--version"]`, since the bare `cargo-<sub>` binary rejects `--version`) and, if absent, its `install` command runs (printing an `Installing` notice). When a cargo tool sets `update = true`, the installed version (parsed from `check`'s stdout) is compared against the latest from its `semver_check` mode (only `crates-io` today, the default, which queries the crates.io API and needs a `crate` name) and re-installed if newer; a failed version check — or an unparseable installed version — skips the update (warns) and keeps the current version. A failed `install` aborts the run. An **os** tool has only `check` (required) plus an optional `install` and `hint`, and no update support: when absent and `install` is declared it runs (like a cargo tool), otherwise the run aborts with `RequiredToolMissing` stating the requirement and any `hint`. Every ensure announces a `Checking` line and prints an outcome (`Present`/`Up to date`/`Installing`/`Updating`), each printed in the same cargo-style status-line format as commands: a right-justified, bold-cyan label prefix in a shared column followed by the tool name and detail.

## Commands

```bash
cargo build                       # build the whole workspace
cargo test                        # run all tests
cargo test -p librake             # test a single crate
cargo test parses_targets         # run one test by name substring
cargo run -p rake -- --list       # run the standalone binary
cargo run -p cargo-rake -- rake --list   # run the cargo-subcommand binary
```

Most lints (the large `deny(...)` blocks, `clippy::pedantic`, rustdoc lints) are gated behind the `nightly` cfg, which `build.rs` sets only on a nightly toolchain. To surface them locally use a nightly toolchain, e.g. `cargo +nightly clippy --all-targets`. The `unstable` feature additionally enables nightly-only `feature(...)` gates; it is a no-op on stable.

## Workspace layout

Four crates, sharing dependency versions via `[workspace.dependencies]`:

- **`librake`** — all the real logic; the two binaries are thin shells over it.
  - `rakefile.rs` — `Rakefile`/`Target`/`Command` model (a target holds a `Vec<Command>`; each `Command` has `name`/`cmd: Option<Vec<String>>`/`sh`/`fish`/`ps: Option<String>`/`skip_on_error`; the `Rakefile` carries only the optional top-level `toolchain` as `Option<String>`, exposed by `toolchain() -> Option<&str>` — there is no top-level shell key). Shell detection lives here: a `ShellFamily` enum (`Posix`/`Fish`/`Ps`) with `key()` (`sh`/`fish`/`ps`), `variant(&Command)` (the matching `Option<&String>`), and `interpreter()` (`("sh","-c")`/`("fish","-c")`/PowerShell `("pwsh"|"powershell","-Command")`, the PS program chosen by `program_on_path`); `classify_shell(name)` maps a basename to a family; `shell_family_from_env(ps_channel, ps_module_nonwindows, shell, is_windows)` is the pure ordered resolver (PowerShell env signals → `$SHELL` basename → platform default) that `detect_shell_family()` feeds from the real env. TOML parsing, validation (every target has ≥1 command **or** ≥1 dependency; `validate_command` requires exactly one *kind* of body — `cmd` mutually exclusive with the shell variants — yielding `MissingCommandBody` (none), `AmbiguousCommandBody` (`cmd` + a variant), `EmptyCmd` (empty `cmd`), or `EmptyShell { variant }` (a blank `sh`/`fish`/`ps`); command `name`s unique within a target; and — only when present — `toolchain` a single non-empty token). Execution: `Rakefile::run` detects the family once (`detect_shell_family()`) and delegates to the public `run_with_family(names, family)` (the test seam, so tests pin a family rather than depending on `$SHELL`); it runs a target after its transitive deps, returning a `RunReport` whose `status` is `Option<ExitStatus>` — `None` when no command ran, i.e. a depends-only target chain — plus the total wall-clock `elapsed`; `run_one(name, target, family)` runs the target's commands in order, timing and printing each command's right-justified `Cmd Runtime`, and stops at the first non-zero exit unless that command sets `skip_on_error`, returning an `(Option<status>, stop)` pair; `spawn_command(target, command, family)` resolves the program + args via `Command::invocation(family)` (a `cmd` body splits into program + args; a shell body becomes the family's interpreter program + flag + the variant line, or `None` → `MissingShellVariant`), prints a blank line then the command's cargo-style status line (`Running [ <name> ] <resolved program + args>`, e.g. `sh -c <line>` or `pwsh -Command <line>`; the command name green via `GREEN` on a TTY, no `====` banners) and spawns it inheriting stdio. `Command::display()` (shell-agnostic, for `--list`) joins a `cmd` or renders each defined variant as `<key>: <line>` joined by ` | `. Output is cargo-style: every status line is a right-justified label prefix in one shared column followed by plain info; command/tool prefixes are bold cyan (matching cargo's status gutter), `Cmd Runtime`/`Runtime` bold green, and every line's info is prefixed with a `[ rake ]` tag (`RAKE_TAG`, color `BOLD_MAGENTA` to match nextest's package-name color) so rake's own output reads apart from the subprocesses it spawns. The column is a compile-time `LABEL_WIDTH` const (currently 12) — the wider of cargo's `CARGO_LABEL_WIDTH` (12, matching cargo's own status gutter) and the longest fixed `STATUS_LABELS` entry (`Running`/runtime labels/tool verbs), so a longer label still widens it automatically; command names live in the info after `Running`, so the width never depends on them. `print_label`/`print_justified` write these lines; `format_duration` renders a `Duration` with tiered µs/ms/s/min units (its integer part space-padded to `INT_WIDTH` = 4 so values line up on the decimal point) and `print_runtime(label, elapsed)` prints a right-justified, bold-green runtime line to stderr (used for the per-command `Cmd Runtime`), while `print_total_runtime(elapsed)` prints the binaries' final `Runtime` line with a bold-green label but a bold-yellow value (`BOLD_YELLOW`) to set the overall total apart. Note `std::process::Command` is imported as `ProcessCommand` to avoid clashing with the model `Command`.
  - `graph.rs` — dependency-graph logic over an `IndexMap<String, Target>`: `validate` (rejects unknown deps and cycles via DFS) and `execution_order` (post-order DFS; deps before dependents, declaration order as tie-break, root last). Multiple roots run as independent, concatenated graphs in the order given (fresh `visited` per root): a target shared across roots runs once per root, while a target reached twice within one root's graph still runs once. (Tools are still ensured once for the whole run — that dedup lives in `Rakefile::run`, not here.)
  - `tool.rs` — the `[tool]` table, modelled by `ToolTable { cargo: IndexMap<String, CargoTool>, os: IndexMap<String, OsTool> }`. `CargoTool` (`crate`/`check`/`install`/`update`/`semver_check`) is the old `Tool`; `OsTool` is `check`/`install`/`hint` (no update). `SemverCheck` enum is only `CratesIo` today. `validate(&ToolTable, targets)` checks cargo tools (non-empty `check`/`install`, `crate` required when `update` under crates-io), os tools (non-empty `check`), name uniqueness across categories (`DuplicateTool`), and that every target `tools` reference resolves against either category. `ToolTable::ensure(name)` dispatches: a cargo name → `ensure_cargo`, an os name → `ensure_os` (a name in neither is a no-op — references are validated up front). `ensure_cargo` (announce a `Checking` line; run `check` capturing output for presence + version; install missing tools inheriting stdio; print a `Present` outcome when present and not updating; `update_if_newer` re-installs (`Updating`) when `latest_version` reports a newer release or prints `Up to date` — an unparseable installed version or a version-check failure skips the update with a non-fatal warning rather than reinstalling). `ensure_os` checks presence, prints `Present`, and on absence either runs a declared `install` (`Installing`) or returns `RequiredToolMissing`. `run_install(name, &[String])` is shared by both (it takes the `install` slice directly). `eprint_tool` builds each tool status line's info (`{name}` plus any `{detail}`) and delegates to `rakefile::print_label` for the right-justified bold-cyan prefix. `latest_crate_version` queries crates.io via `ureq`; `parse_version_token` pulls a semver out of `check`'s stdout. The `print_label` helper is reused from `rakefile.rs` (made `pub(crate)`).
  - `toolchain.rs` — `ensure_rust_toolchain(toolchain: Option<&str>)`, called by **both** binaries. `None` (no declared channel) is a quiet no-op when `cargo` is present and only bootstraps `stable` when `cargo` is entirely absent. `Some(ch)` probes `cargo --version`; bootstraps Rust via the rustup installer when absent (`install_rustup`, then `prepend_cargo_bin_to_path`); else classifies the channel against `rustup toolchain list` (`toolchain_status`/`toolchain_listed`) and `rustup toolchain install <ch>`s a `Missing` one; then `pin_toolchain` sets `RUSTUP_TOOLCHAIN`. An `Unmanaged` result (no rustup) accepts the existing cargo. Installs require an interactive stdin (`affirmative` parses the `[y/N]` `prompt`); `set_env` wraps the edition-2024 `unsafe` `std::env::set_var` (safe pre-thread-spawn) behind a scoped `#[allow(unsafe_code)]`. Reuses `rakefile::print_label` for status lines.
  - `error.rs` — `Error` enum (`thiserror`) and the crate `Result` alias. Tool failures add `UnknownTool`, `EmptyToolCommand`, `ToolUpdateMissingCrate`, `ToolInstallSpawn`, `ToolInstallFailed`, `RequiredToolMissing` (an absent os tool with no `install`), `DuplicateTool` (a name in both `[tool.cargo]` and `[tool.os]`); command-body validation adds `MissingCommandBody` (no body), `AmbiguousCommandBody` (`cmd` mixed with a shell variant), `EmptyShell { variant }` (a blank `sh`/`fish`/`ps`), and the run-time `MissingShellVariant` (the detected shell has no matching variant); toolchain handling adds `RustToolchainMissing`, `RustInstallSpawn`, `RustInstallFailed`, `RustChannelMissing`, and `InvalidToolchain`.
  - `cli.rs` — the shared clap `Cli` derive (`file`/`list`/`targets`) and `command(name, bin_name)` helper (`pub mod cli`). Both binaries and `xtask` build their `clap::Command` from here so the CLI cannot drift.
  - `lib.rs` — public re-exports (incl. `CargoTool`/`OsTool`/`ToolTable`/`SemverCheck`, `ShellFamily`/`detect_shell_family`, `ensure_rust_toolchain`), the `pub mod cli`, `DEFAULT_TARGET` const, `exit_code`, `list_targets` (shows `tools:` lines), `format_duration`, `print_runtime`.
- **`cargo-rake`** — the `cargo rake` subcommand binary.
- **`rake`** — the standalone `rake` binary. Not published to crates.io (the `rake` name is taken by an unrelated crate); `publish = false`. Distributed via AUR/DEB/RPM/Homebrew instead.
- **`xtask`** — workspace automation (`publish = false`). `cargo xtask dist rake` generates the packaging sidecars (man page via `clap_mangen`, bash/zsh/fish completions via `clap_complete`, license files, example `Rakefile.toml`) into `dist/rake/`, reusing `librake::cli::command`.

`cargo-rake` and `rake` are near-identical clap front-ends, both parsing the shared `librake::cli::Cli` via `librake::cli::command(...)` (layering their own `version`/`long_version`). Both call `librake::ensure_rust_toolchain(rakefile.toolchain())` (after the `--list` short-circuit, so `--list`/`--help`/`--version` skip it) and both surface a failed run via its chained `Display` (`{err:#}`) + `exit(1)` rather than anyhow's default `Error: {Debug}`. The only real difference: `cargo-rake::main` strips a leading `rake` argument, because Cargo invokes subcommands as `cargo-rake rake ...` (and, incidentally, only `rake` realistically reaches the bootstrap branch, since `cargo rake` always runs through an existing cargo). Keep their `run` logic in sync when changing one; the argument definitions live only in `librake::cli`.

Packaging/distribution lives in `packaging/` (AUR PKGBUILDs under `arch/`, nfpm DEB/RPM under `nfpm/`, Homebrew template under `homebrew/`) and the `release.yml` / `test-aur-publish.yml` workflows. `cargo-rake` ships via crates.io + `cargo binstall` (`[package.metadata.binstall]`); `rake` ships as four AUR packages (`rake`, `rake-unstable`, `rake-bin`, `rake-unstable-bin`), DEB/RPM, and Homebrew. The `-unstable` variants build with `--features unstable` on the stable toolchain — that feature is a functional no-op (it only toggles nightly-only lint gates), so no nightly toolchain is involved.

## Conventions

- **No `unwrap`/`expect` and no panicking operations anywhere, including tests.** Use `match`/`ok_or`/`?` and return `Result` instead. Tests use `Result<(), Box<dyn Error>>` and convert failures with `.into()`/`Err(...)`. This is enforced by intent, not just lints — see the deliberately-spelled-out `exit_code` in `lib.rs`.
- A command that *runs but exits non-zero* is not an `Error`; its `ExitStatus` propagates so the caller chooses the process exit code (`exit_code` maps a signal-killed child to `1`). Reserve `Error` for load/parse/validation/spawn failures.
- Target order is significant and preserved with `IndexMap` (`toml` uses `preserve_order`); don't swap in a `HashMap`.
- Each crate's `build.rs` uses `vergen-gix` to emit build/git/rustc/system info, rendered by `vergen-pretty` as the binaries' `--version` long output.
- Edition 2024, MSRV 1.95.0.
